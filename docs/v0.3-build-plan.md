## Goals & Non-Goals

**Goals.** Add 22 novelty UX features that each amplify one of three UX pillars (programmable links, real-time visible money, composable identity), serve BOTH agent and consumer flows, and ship purely on DevNet. Build 4 new on-chain primitives that compose with existing Pact architecture.

**Non-goals.** No creator-economy commerce surfaces (storefront CMS, calendar booking, newsletter, paid subscription content, affiliate splits, wishlist, tax export, yield, DCA, generic marketplace). Each one of those would dilute the agent-payment thesis and put us in a head-on race with Whop / Gumroad / Patreon / Calendly. Settle's wedge is programmable agent + human payments, not creator commerce. We let agent-payment depth drive distribution.

---

## The Three UX Pillars

Every feature in the plan amplifies at least one. If a proposed feature doesn't, it doesn't ship.

**A. Programmable links.** Every action surface (pay, request, fund pact, hire agent, revoke) is a Blink. URLs are transaction surfaces. They render natively in X via Phantom's Blink renderer and Dialect Actions Registry.

**B. Real-time visible money.** Solana's <1s finality stops being internal nicety and becomes UX. Confetti scaling, sub-400ms confirm animations, live audience counters, streaming spend timers visible to both sides.

**C. Composable identity.** Receipts are live objects that update post-purchase. Voice notes attach. Follows live in wallet. Reputation portable across apps via on-chain attestations.

## New Architectural Primitives (build first; everything else depends on these)

### P1. Streaming Pact (new Anchor instruction set)

Rationale: Codex's strongest single suggestion. The pay-as-you-go primitive is the natural extension of our existing Pact + Vault PDA. We don't use Streamflow because rolling our own composes cleanly with the existing PDA + cap enforcement we already have, and lets us preserve the cross-pact daily-cap invariant we just shipped.

**State change** (`programs/settle-agent-card/src/state.rs`):

```rust
pub enum PactMode {
    OneShot { spent: u64, cap_lamports: u64 },
    Streaming {
        rate_lamports_per_slot: u64,
        max_total_lamports: u64,
        claimed: u64,
        last_claim_slot: u64,
        paused: bool,
        pause_accumulated_slots: u64,
    },
}

#[account]
pub struct Pact {
    // existing fields stay (parent_card, authority, agent_pubkey, scope_label_hash, usdc_mint,
    //                      allowlist, expiry_slot, closed, created_at, bump, vault_bump)
    pub mode: PactMode,
}
```

`Pact::SPACE` grows to accommodate the bigger enum. `OneShot` carries the existing `cap_lamports` + `spent`; `Streaming` carries the new fields.

**New instructions** (in `instructions/`):

- `open_streaming_pact(rate_lamports_per_slot, max_total_lamports, allowlist, expiry_slot)` — same shape as `open_pact` but stores `Streaming` mode and funds vault with `max_total_lamports`. Authority signs. Atomic.
- `claim_streaming(merchant_owner, capability_hash, receipt_hash, reason_hash, policy_snapshot_hash)` — agent signs. Computes entitlement = (current_slot - last_claim_slot - pause_accumulated_slots) * rate_lamports_per_slot, capped by max_total - claimed. CPIs TransferChecked from vault → merchant. Updates claimed, last_claim_slot, card.used_today (cross-pact daily cap stays enforced). Emits PolicyDecision + new PactStreamClaimEvent.
- `pause_streaming(pact)` / `resume_streaming(pact)` — authority-only. Sets paused flag and tracks accumulated paused slots so future claims don't bill for paused time.
- `close_pact` — extended to handle Streaming mode (refund vault.amount = max_total - claimed).

**Files touched:** `state.rs`, `events.rs`, `errors.rs`, `instructions/{open_streaming_pact, claim_streaming, pause_streaming, resume_streaming, close_pact}.rs`, `lib.rs`. SDK: `idl.ts`, `pdas.ts` (no new PDA — shares Pact seed). Web: `lib/anchor-client.ts` (new ix builders), `lib/account-decoder.ts` (decode the enum mode).

**Tests:** 4 new integration tests:
- open_streaming → claim at t+10s → verify accumulated correctly
- pause → wait 10s → resume → claim → verify paused-slot accounting
- claim past max_total → fail
- close_pact on streaming → refund correct unspent amount

### P3. Voice-Note Receipt Attachments

Rationale: turns receipts into emotional artifacts (#15). Sealed-box encrypted, only recipient decrypts.

**Off-chain only.** Supabase Storage bucket `receipt-attachments`, RLS-gated.

**Rights model decision (closed):** the attachment's decrypt-rights are pinned to the **original receipt recipient pubkey** (`receipts.card_pubkey → agent_cards.authority_pubkey` for agent receipts; `merchant_pubkey` for merchant-side receipts). Transferring the receipt cNFT to a third party does NOT transfer decrypt rights. This is honest and avoids the key-rotation-on-transfer problem that V1 isn't ready to solve. Transferable rights bundles are explicitly deferred to V0.4 (would require either a viewing-key model with on-transfer re-encryption, or an on-chain rights NFT separate from the collectible cNFT).

**New table** (migration `0007_receipt_attachments.sql`):
```sql
create table public.receipt_attachments (
    id          uuid primary key default gen_random_uuid(),
    request_id  uuid not null references public.receipts(request_id) on delete cascade,
    kind        text not null check (kind in ('voice_note', 'text_note', 'image')),
    storage_path text not null,                 -- Supabase Storage object path
    sealed_box_for_pubkey text not null,        -- pubkey the blob is sealed to (the receipt recipient)
    duration_ms integer,
    mime_type   text,
    created_at  timestamptz not null default now()
);
create index ra_request_idx on public.receipt_attachments (request_id, created_at desc);
alter table public.receipt_attachments enable row level security;
-- Read-access stays with the original receipt recipient. cNFT transfer does NOT widen this.
create policy ra_recipient_read on public.receipt_attachments for select using (
    sealed_box_for_pubkey = coalesce(auth.jwt()->>'wallet_pubkey', '')
    or request_id in (
        select r.request_id from public.receipts r
        join public.agent_cards c on r.card_pubkey = c.card_pubkey
        where c.authority_pubkey = coalesce(auth.jwt()->>'wallet_pubkey', '')
    )
);
```

**New endpoints:**
- `POST /api/receipts/[requestId]/attachments` — wallet-sig auth, accepts multipart upload, sealed-box encrypts client-side, server stores ciphertext in Storage and metadata in table
- `GET /api/receipts/[requestId]/attachments` — wallet-sig auth, returns signed URLs for ciphertext blobs (auth.pubkey must be the sealed_box_for_pubkey or the original card.authority)
- `POST /api/receipts/[requestId]/attachments/decrypt-key` — wallet-sig auth, returns the sealed-box envelope so the client can decrypt locally

### P6. Jupiter Swap-and-Fund (DevNet caveat)

Rationale: "Pay with what's in my wallet" — buyer holds BONK, JUP, USDT, anything. Jupiter swaps to USDC under the hood. Recipient gets clean USDC.

**DevNet reality:** Jupiter Lite API (`https://lite-api.jup.ag/swap/v1/quote` and `/swap-instructions`) is mainnet-focused. DevNet has no real Jupiter liquidity. Three options:

1. **Mainnet quote, devnet token mock** — get quote from Jupiter mainnet (read-only), surface UX, then on devnet just route directly through our test USDC. Works for demo, not for real swap.
2. **Skip DevNet swap** — show the UI in a "preview mode" that says "swap will activate on mainnet."
3. **DevNet liquidity pools via Orca** — if we deploy our own test pools, real swaps work. Real engineering, but worth it for demo authenticity.

**Decision:** ship option 1 for DevNet (quote real, swap mocked through a USDC route). Document clearly in the UX. Ship option 3 for mainnet.

**New endpoint:** `POST /api/swap/quote-and-build`
- body: `{ inputMint, outputMint: USDC, amount, mode: 'send' | 'fund_pact' | 'fund_streaming_pact', recipientArgs }`
- For DevNet: returns a tx that does USDC→USDC routing with a synthetic "swap completed" receipt event for the UI
- For mainnet: returns Jupiter swap-instructions composed with our send/fund ix into a single signed tx

**Files:** `apps/web/lib/jupiter.ts` (new), `apps/web/app/api/swap/quote-and-build/route.ts` (new).

### P8. Public Capability Leaderboard (with real latency)

Rationale: this is the recombination win from earlier (Strava × capability hashes). Agents compete on real measurable performance per capability_hash. **The audit data IS the marketing surface.**

**Latency metric — fix from codex review.** The earlier metric (`created_at - to_timestamp(decision_slot * 0.4)`) mixed wall-clock with hardcoded slot-time and could go negative under clock drift. Replaced with a server-clock-consistent measure based on real start/end timestamps captured by the proxy (P10 below).

**Migration `0009_request_timing.sql`** (added before this view):
```sql
alter table public.receipts
    add column if not exists request_initiated_at  timestamptz,  -- proxy entry, before any RPC roundtrip
    add column if not exists upstream_called_at    timestamptz,  -- merchant call started
    add column if not exists upstream_returned_at  timestamptz;  -- merchant call returned
create index if not exists receipts_capability_idx
    on public.receipts (capability_hash, decision, public_feed)
    where decision = 'ALLOW' and public_feed = true;
```

The x402 proxy populates these in `persistReceipt` (already centralized — single edit point in `apps/web/app/api/x402/proxy/[merchant]/route.ts`).

**New view** (replaces the slot-based math entirely):
```sql
create or replace view public.capability_leaderboard as
select
    capability_hash,
    merchant_pubkey,
    count(*)::int as completed,
    -- end-to-end: time from buyer/agent request to settled receipt persistence
    avg(extract(epoch from (created_at - request_initiated_at)) * 1000)::numeric(10,1)
        filter (where request_initiated_at is not null) as avg_total_latency_ms,
    -- merchant call only: isolates the upstream service's responsiveness
    avg(extract(epoch from (upstream_returned_at - upstream_called_at)) * 1000)::numeric(10,1)
        filter (where upstream_called_at is not null and upstream_returned_at is not null) as avg_merchant_latency_ms,
    avg(amount_lamports)::bigint as avg_amount_lamports,
    sum(amount_lamports)::bigint as total_volume,
    count(distinct card_pubkey)::int as unique_users,
    max(created_at) as last_used_at
from public.receipts
where decision = 'ALLOW' and public_feed = true
group by capability_hash, merchant_pubkey;
```

Both latencies are server-clock consistent (same proxy process). `avg_merchant_latency_ms` is the defensible metric for "how fast is this merchant" — it isolates the upstream service from facilitator overhead. Pre-migration receipts have NULL timing fields and are excluded from the average (the `filter` clause handles this honestly).

**New page:** `/leaderboard/[capabilityHash]` shows ranked merchants. `/leaderboard` index lists top capability hashes.

**New endpoint:** `GET /api/leaderboard/[capabilityHash]`.

### P9. Delivery Escrow Pact (new on-chain primitive)

Rationale: the existing `spend_via_pact` transfers funds **straight to the merchant**. There is no escrow hold and no dispute window. F22 (buy-now-pay-on-delivery) and the buyer-side dispute flow in F4 were not actually buildable on the v0.2 architecture; they need a real state machine.

**State change** in the same `PactMode` enum that gains `Streaming`:
```rust
pub enum PactMode {
    OneShot { spent: u64, cap_lamports: u64, single_use: bool, consumed: bool },
    Streaming { ... },
    DeliveryEscrow {
        amount: u64,
        merchant: Pubkey,
        capability_hash: [u8; 32],
        confirm_deadline_slot: u64,    // after this, anyone may release to merchant (no buyer-confirmation needed)
        dispute_deadline_slot: u64,    // before this, buyer may dispute (refund); after, dispute is closed
        released: bool,
        refunded: bool,
    },
}
```

**New instructions:**

- `open_delivery_escrow(amount, merchant, capability_hash, confirm_deadline_slot, dispute_deadline_slot)` — buyer (= card.authority) signs. Funds vault with `amount`. `confirm_deadline_slot` typically 7 days out; `dispute_deadline_slot` matches.
- `release_delivery_escrow` — two valid callers:
  1. Buyer (card.authority) signs to confirm receipt early. Transfers vault → merchant ATA.
  2. Anyone (permissionless) calls after `confirm_deadline_slot`. Transfers vault → merchant ATA. This is what protects the merchant from a silent buyer.
  Both paths require `!released && !refunded`.
- `dispute_delivery_escrow` — buyer signs before `dispute_deadline_slot`. Transfers vault → buyer's USDC ATA. Sets `refunded = true`. Distinct from `close_pact` because it has the timing constraint and emits a different event for the unified ledger.

**Constraints (all enforced on-chain):**
- Merchant pubkey is pinned at open; release cannot redirect.
- `confirm_deadline_slot ≤ dispute_deadline_slot` enforced at open.
- After release: only refund-on-dispute window has expired; release is permissionless.
- Anyone calling release after deadline pays the tx fee themselves; merchant doesn't have to maintain a fee-paying wallet (or can run a cron that calls it for free).

**Files touched:** `state.rs` (extend enum), `instructions/{open_delivery_escrow, release_delivery_escrow, dispute_delivery_escrow}.rs` (new), `lib.rs`. SDK + web client + decoder mirror.

**Tests** (in addition to the existing 12):
- Buyer confirms within window → merchant receives funds, vault drained
- Buyer disputes within window → buyer receives refund
- Buyer ignores → after `confirm_deadline_slot`, anyone (permissionless) calls release → merchant receives funds
- Buyer tries to dispute after `dispute_deadline_slot` → fails
- Double-release → fails

**F22. Buy-now-pay-on-delivery escrow** (#37) — `consumer`. Built on the new **DeliveryEscrow Pact mode (P9)**, not the existing Pact flow. The current `spend_via_pact` transfers funds directly to the merchant — no escrow hold. P9 introduces a real state machine: open → released | refunded | timeout-released.

UX flow:
1. Buyer taps "Buy with delivery protection" on a product. Wallet signs `open_delivery_escrow(amount, merchant, capability_hash, confirm_deadline_slot=now+7days, dispute_deadline_slot=now+7days)`.
2. Funds sit in the Pact Vault PDA. Merchant ships.
3. Buyer taps "I received it" within 7 days → on-chain `release_delivery_escrow` runs, vault drains to merchant.
4. Buyer disputes within 7 days (taps F4's 😞) → on-chain `dispute_delivery_escrow` runs, vault drains back to buyer.
5. Buyer never confirms and never disputes → on day 8, anyone (a permissionless cron we run, or the merchant themselves) calls `release_delivery_escrow` → merchant gets paid.

Touch: program (P9) — already specified. New endpoints: `app/api/pacts/[id]/escrow/release/route.ts`, `app/api/pacts/[id]/escrow/dispute/route.ts`. Add a permissionless cron worker in `apps/indexer/src/escrow-cron.ts` that polls for expired escrows and calls release. UI: `app/cards/[id]/page.tsx` shows the right action button per state. Depends on: **P9**. Effort: ~3 days after P9 lands.

**Codex round-2 fix (acknowledged):** the earlier framing assumed F22 was a UI wrapper over the existing close_pact flow. It's not — close_pact is authority-only and doesn't have the merchant/timeout state machine. F22 needs P9 as a distinct on-chain primitive.

## Codex Round-2 Open Questions — Answered

1. **F5: cNFT transfer = collectible only, or rights bundle?**
   → **Collectible only.** Refund-rights and decrypt-rights stay pinned to the original receipt recipient pubkey via Supabase RLS. Rights-bundle transfer is V0.4 and requires a viewing-key model with on-transfer re-encryption (or an on-chain rights NFT separate from the collectible). The UI is honest about this.

2. **F22: new escrow Pact mode, or UI wrapper over existing close/release?**
   → **New on-chain Pact mode (P9 DeliveryEscrow).** Existing `spend_via_pact` transfers funds straight to the merchant — no escrow hold, no release/dispute window. F22 needs a real state machine: `open → released | refunded | timeout-released`. Specified in P9 with merchant pubkey pinned at open, dispute window, permissionless post-deadline release.

3. **P3-P8 (now P3-P10): silent prerequisites or scheduled?**
   → **Scheduled into the wave that introduces them.** Each wave's row in the sequencing table explicitly names the primitive it builds. Effort grew from 12 to 14-15 weeks once the primitive cost is honestly absorbed.

## Codex Review Checklist

When codex reviews this plan, the questions to answer are:

1. **Is each feature on-thesis?** Every one of F1–F22 amplifies one of the three UX pillars and applies to either agent flow, consumer flow, or both. Anything that doesn't = cut.
2. **Does each feature actually use a Solana primitive that can't easily be done on EVM?** Streaming Pact (sub-second slot accounting), capability leaderboard (audit data as marketing surface), Blink-everywhere (Phantom in-Twitter render), sub-400ms confetti (chain-finality-driven UX), pay-with-any-token (Jupiter), self-repricing QR (Solana Pay reference). Yes.
3. **Are the 4 new on-chain primitives (P1-P2) audit-shaped?** P1 (streaming) extends existing Pact + Vault; same auth model. P2 (single-use) is a 2-bit state addition. Both compose with the cross-pact daily-cap fix that landed in v0.2 round-2.
4. **Is anything overlooked in the codebase?** Verify: SAS, Lighthouse, Squads, Token-2022 confidential transfers — codex should flag if any of those would compose better than what we have.
5. **DevNet feasibility:** Jupiter is the one caveat. Everything else fully on devnet.
6. **Sequencing:** does the wave order respect dependency? P1 must precede F13. P3+P4 must precede F5. P5 must precede F8. Otherwise parallel.
7. **Effort sanity check:** 12 weeks for 22 features + 4 primitives. Roughly: ~3 days/feature average, with bigger features (F5 receipt-as-live-object, F12 Jupiter, F13 streaming) eating 5-7 days each.
8. **Anything missed from the recombinations** (Twitch × receipts, SoundCloud × streaming, Apple Wallet × Pact)? Capability leaderboard is the strongest one and it's in (F17 + P8). Apple Wallet pass we defer (would be V0.4 mobile).

## Summary for the User

This plan ships **22 novelty UX features** (curated from the 40-list with the 15 off-thesis ones explicitly cut) plus **4 new on-chain primitives** that compose with our existing v0.2 architecture. Every feature serves both AI-agent flow and consumer flow where applicable. Every feature has a Solana primitive backing it. Everything is buildable on DevNet today, with the single documented caveat that Jupiter swap demos in mock-mode on DevNet (real on mainnet).

**The signature UX in this plan is F5 — the live receipt that talks back.** That's the single feature that makes Settle feel uncanny: receipts as live objects with refund timers, voice notes, transferable warranties, follow-up notifications. Nothing else in the payment-app space has it.

**The architectural extension is F13 — Streaming Pact.** That's what makes the AI-agent payment thesis literal: agents earn salary per-second, authority can pause/cancel/refund instantly. This is the moat-deepening primitive.

**The viral mechanic is F8 + F9 + F10 — handle-as-Venmo-request, self-repricing QR, one-time-use links.** These compound: every pasted link is a Blink, every QR auto-prices, every pre-authorized link is single-use.

Ready for codex review.
